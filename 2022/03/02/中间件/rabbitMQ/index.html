MQ

相关概念

什么是MQ

  • message queue,本质是一个先入先出的队列(FIFO).
  • 队列的内容是message
  • 是一种跨进程的通讯机制,作用于上下游传递消息.
  • 在互联网框架中,MQ是一种非常常见的上下游用于”逻辑解耦+物理解耦”的消息通讯服务. 消息发送上游只需要依赖Mq, 不依赖其他服务.
    • 上游将消息放在mq, 下游订阅mq数据, 对上游没有影响.

      使用MQ原因

  • 流量削峰
    • 实际业务中做请求的缓存处理.
      • 比如请求处理速度是10000次/s, 正常处理情况下每秒超过10000就不可以处理了. 但是如果加入mq缓存后,将请求排队处理,用户可能需要等待一段时间才可以得到反馈结果. 防止服务器宕机
  • 应用解耦
    • 上游系统和下游系统关联性强的话, 下游的故障会影响上游的操作
      • 比如 订单系统需要调用物流系统, 当物流系统故障后, 订单系统将无法响应. 但加入mq后, 待物流系统回复后依旧会执行订单系统在mq发布的message信息, 不影响业务运行.
  • 异步处理
    • 进行一个异步响应的功能,不再影响主线程
      • 比如 之前a调用b的方法, a需要提供一个callback方法,在b执行结束调用callbck时, a才知道b调用结束. 现在使用消息总线, b调用后将结果交给mq, a订阅了mq的消息皆可以实现异步响应

        MQ分类

  • activeMQ
    • 优点:万级吞吐量, 时效性ms,主从框架实现可用性, 很难丢失消息
    • 缺点:社区维护越来越少
  • kafka
    • 优点:百万级吞吐,性能卓越,时效性ms,分布式形式一个数据多副本可以处理宕机问题,保证所有的消息只被消费一次,具有优秀的后台web界面, 主要支持简单的mq功能,在大数据领域用于实时计算以及日志采集.
    • 缺点:单机超过64队列/分区,load会明显飙高(cpu),队列越多, load越高, 发送消息响应越久. 消息失败不可以重试, 一台代理宕机会消息乱序,社区更新慢.
  • rocketMQ
    • 出自阿里,java编写
    • 优点:单级吞吐量十万级, 消息0丢失, 分布式扩展性好, 支持10亿消息堆叠, 源码java可读性强,可定制
    • 缺点: 支持的客户端语言不多,目前java和c++,但c++不成熟;社区活跃一般
  • rabbitMQ
    • 基于AMQP(高级消息队列协议),主流的消息中间件之一
    • 优点:erlang语言实现高并发, 性能好; 吞吐量万级, MQ功能比较完备,支持多种语言.具有管理界面 社区活跃
    • 缺点:学习成本高,商业版收费

      MQ选择

  • kafka
    • 大型公司具有日志采集功能,需要大数据处理
  • rocketMQ
    • 天生为金融互联网领域而生, 可靠性强, 多应用于电商业务中, 处理多并发情况
  • rabbitMQ
    • 结合erlang并发特性,性能好时微秒级,社区活跃,数据量不大,中小型公司的选择

      rabbitMQ

  • 消息中间件, 主要功能接收,存储,转发消息.

    四大核心概念

  • 生产者: 产生数据发送消息的程序, 上游
  • 队列: 属于mq内部结构
  • 交换机: 属于mq内部结构, 一个交换机可以对应多个队列(实现绑定的关系).
  • 消费者: 消费数据的程序, 下游, 和队列一一对应

    名词解释

  • virtual host: 处于多租户和安全因素设计, 把amqp基本组件划分到一个虚拟的分组中, 类似于网络中的namespace, 多个不同用户使用同一个rabbitmq server 提供服务时, 会划分出多个vhost, ,每个用户在自己的vhost创建exchange/queue
  • producer: 生产者
  • connection: 连接 包含多个信道, 是生产者和消费者和broker之间的tcp连接
  • channel: 信道, 每个生产者和mq之间有对应的信道, 用为防止每个tcp连接造成的极大消耗. 在一个connection内部有多个信道,信道作为通讯的最小单位, 每个行到有自己的标识 channel id 用于客户端和mq的识别. 所以说通讯的channel是轻量级的, 他极大的减少了操作系统在tcp连接上的开销
  • broker: 接收和分发信息的应用
  • exchange: 交换机 message到达broker的第一站, 根据分发规则, 匹配表中routing key分发消息到queue中.
  • queue: 队列
  • consumer: 消费者
  • binding: 用于标识交换机和队列之间的虚拟连接本身包含routing key, 该信息保存在交换机的查询表中, 用于message分发依据

        消息应答问题

  • 消息应答问题

    • 为了防止消息丢失(消费者完成了某个消息的部分就宕机) ,rabbitmq引入了消息应答机制,消息应答就是:消费者在接收到消息并且处理该消息之后,告诉rabbitmq它已经处理了,rabbit可以把该消息删除
    • 自动应答
      • 需要在高吞吐量和数据安全性方面做权衡
      • 不在乎消费者对消息处理是否成功，都会告诉队列删除消息。
    • 手动应答
      • 消费者处理完业务逻辑，手动返回ack（通知）告诉队列处理完了，队列进而删除消息
      • channel.basicAck(var1 var2) 确认消费成功
        • var1 消息
        • var2 是否批量应答
      • channel.basicNack(var1 var2) 消费失败
        • var1 消息
        • var2 是否批量应答
      • channel.basicReject(var1) 消费拒绝(不再考虑该消息直接丢弃)
        • var1 消息
      • 批量应答
        • 当队列上有5,6,7,8四个消息时, 如果批量应答是true 则同时应答5,6,7,8. 如果是false则只应答8

        持久化问题

  • 持久化问题

    • 队列持久化(生产者行为)
      • mq宕机该消息队列会被持久在磁盘中
      • 创建持久化队列, 注意已经创建的队列不可以重复声明, 只有新创建或者删除原队列后才可以使用.
    • 消息持久化(生产者行为)
      • mq宕机后消息会被持久化到磁盘, 但是不是绝对的不丢失
    • 不公平分发(消费者行为)
      • 默认是轮询分发, 但是存在不公平分发的情况, 处理效率高的消费者应该跟多的机会处理消息, 效率低的消费者处理的机会更少. 一方面防止消费者闲置, 一方面提高整体处理效率.(遵循能者多劳)
      • 按照上述案例添加不公平分发, 结果work03执行的多, work04执行的少
    • 预取值(消费者行为)
      • prefetch相当于在轮询分发的基础上加上权重,
        • 如默认按顺序给work01,work02分发一条消息, 设置work01的prefetch=2后, 按照给work01分发两条消息,给work02分发一条消息.
      • 相当于channel作为一个缓冲区存放数据, 定义通道上允许的未确认消息的最大数量
        • prefetch=5 就表示channel只可以堆积五条数据, 再来的数据就不会进入channel
      • 注意: 不公平分发相当于预取值=1, 预取值实现的基础在不公平分发上, 在平等竞争的基础上, 依旧是效率高的执行机会多

        publisher Confirms(发布确认)

  • 在生产者实现队列持久化, 消息持久化的基础上, 有可能存在进行磁盘写入的过程中宕机的现象.

    • 需要mq向生产者再次确认

      发布确认的策略

  • 开启发布确认的方法

    • 由生产者创建的信道执行此方法

      单个确认发布

  • 同步确认发布 , 也就是说在一个消息发布并确认后才可以发布下一个信息

  • 缺点发布速度慢, 提供每秒不过百条的吞吐量

    批量发布确认

  • 在一组队列发布后, 进行一次确认, 提高确认的速度

  • 缺点如果某组其中的消息发送错误, 系统无法得知是哪个消息错误, 只会提示该组消息错误

    异步批量确认

  • 消息生产者将消息发布在channel之中无需进行等待, 消息是以Map<序号,内容>的形式存放, broker自动接收消息后会根据情况调用消息的ackCallback和nackcallback方法, 用来告知生产者

  • 执行前信道需要添加监听器addConfirmListener(ConfirmCallback var1, ConfirmCallback var2)

    • var1 消息确认成功的回调函数
    • var2 消息确认失败的回调函数

      如何处理异步未确认消息

  • 已经确认的消息不需要处理, 而未确认的消息多数情况下需要持久化或者是重新发送等操作. 所以后期需要处理下未确认的消息, 这样的话就需要让主线程和监听线程进行一个交互.

  • 使用一个基于内存的能够被发布线程访问的队列, 比如ConcurrentSkipListMap 该队列在confirm和callback之间消息传递 或者是使用redis或者数据库直接持久化

  • 在以上异步批量的基础上修改

      topics(主题模式)

  • 已知情况下可以捆绑多个队列和routingkry

    • 但是在路由已经确定的情况下, 不可能同时访问两个队列和routingkey都不同信道

  • 使用主题模式的交换机功能可以覆盖fanout(发布订阅)和direct(路由模式)

    topics要求

  • 主题模式的交换机必须是一个单调的列表, 以[.]分隔开. 例如 :”stock.user.abd”,”com.alibaba.demo”等, 最多不可以超过255个字节

  • 包含一下可用通配符

    • [*]可以代替一个单词 比如” *.abc.* “
    • [#]可以代替0个或者多个单词 比如” abc.# “

  • 注意:

    • 当一个队列绑定的建是[#], 那么这个队列将接收所有的数据, 类似于交换机的fanout类型
    • 当一个队列绑定的建中没有#或者*, 那么这个队列类似于交换机的direct类型

      死信队列

  • 特定原因导致的无法被消费的消息称为死信

  • 主要应用在消息在消费的过程中出现了异常, 异常的消息会被放在死信队列进行下一步处理(重新消费,写入等操作)

  • 应用场景, 在用户商城下单后一定时间内没有支付, 支付时间失效

    死信的来源

  • 消息ttl过期

  • 队列达到最大长度

  • 消息被拒绝(basic.reject()或basic.nack())并且requeue=false

    • requeue=false 队列拒绝再重新放回

  • 在由上述原因产生的死信, 将手动通过制定交换机将信息放在某个队列中, 该队列就是死信队列

  • 测试实战需要生成两个交换机normal_exchange和dead_exchange, 两个队列normal_queue和dead_queue,选择不同的routingkey 服务于两个消费者consumer01和consumer02

    ttl产生死信

  • 消费者一

    • 完成两个交换机,两个队列
    • 确定消息成为死信后转发到第二个交换机
    • 队列声明参数
      • Map<String, Object> arguments
        • x-message-ttl 队列消息过期时间 本次选择在生产者设置消息过期时间
        • x-dead-letter-exchange 消息重定向的交换机地址
        • x-dead-letter-routing-key 死信交换机对应队列的routingkey

  • 生产者

    • 设置消息过期时间 new AMQP.BasicProperties().builder().expiration(“10000”).build();

    • 在生产者启动后 10s内启动消费者一也会消费所有消息

      延迟队列

  • 在死信队列实例中, 存在ttl情况造成死信, 如果不存在消费者一, 势必所有消息都会过期成为死信, 每个死信都在normal_queue中停留了10s. 这一过程的运用可以称为延迟队列.

    延迟队列场景

  • 场景

    • 订单十分钟未支付自动取消
    • 新建店铺十天没有上传商品, 自动消息提醒
    • 用户注册后三天没登录,短信提醒
    • 用户退款三天没有处理,联系相关运营
    • 预约会议在开始前十分钟通知所有人

  • 以上的所有场景都可以采用定时器替换, 但是如果短时间内有大量的数据比如电商节的订单支付,这时如果还采用轮询的定时器将无比耗费性能,大概率不能在短时间内完成所有订单的状态检查, 会给数据库很大的压力.

幂等性

  • 用户对同一操作发起的一次请求或多次请求的结果一致, 不会因为多次点击而产生副作用, 出现消息重复消费
  • mq中 broker已经将消息发给消费者, 但是消费者在消费后的ack时网络中断, 故mq是没有收到确认的, 就会将消息重新入队消费

    解决方案

  • 使用全局id, UUID或者是其他方案给消息生成一个唯一表示, 消费钱判断该消息是不是已经消费过, 也可以使用mq的id判断
  • 业界主流的幂等性主流解决方案是,
    • 唯一ID+指纹码(一些规则或者时间戳嫁加别的服务给到的唯一信息码),数据库主键去重
      • 通过业务拼接保证该码唯一性, 之后判断是否在数据库中
    • redis原子性
      • 利用redis执行setnx命令,天然有幂等性, 从而实现不重复消费

      惰性队列

  • 消息保存在内存中还是在磁盘下
    • 正常情况: 消息是保存在内存中
    • 惰性队列: 保存在磁盘中
  • 使用在消费者宕机,关闭等(长时间不能消费造成堆积的情况), 使用惰性队列将消息保存, 用于节约内存

    两种模式

  • 队列声明中的参数x-queue-mode的参数
    • default默认模式
    • lazy惰性队列模式

      内存开销

  • 在一百万条消息下, 每条消息1kb, 普通队列占中1.2g内存, 惰性队列因为消息存在磁盘所以只占用1.5m内存
